# LHC v0.3 — pre-registered decision table

**Locked:** 2026-05-12 (before any v0.3 sweep run; before authoring of v0.3 held-out tasks).
**Source of these thresholds:** three rounds of external review across 2026-05-11 and 2026-05-12, accepted in full. See [`../journal/2026-05-12-v03-framing-and-reviewer-lock.md`](../journal/2026-05-12-v03-framing-and-reviewer-lock.md).
**Why pre-registered:** v0.2 taught us that goalpost-adjustment after seeing data is the failure mode. This document is the v0.3 contract.

> **Do not edit this file after the v0.3 Phase 1 sweep starts.** If a threshold turns out to be wrong, write a new decision document for v0.4 with a `Reason for revision` section. The old contract stays in place as the historical record.

## 1. Thesis

Long-horizon coherence failures in agent systems are often **state-interface failures**, not capability failures. v0.3 isolates how state representation, resume protocol, and inconsistency handling change model behavior under delayed/distractor pressure.

We do **not** claim "harness > scale" as a discovery — recent work (Meta-Harness, Natural-Language Agent Harnesses, Externalization in LLM Agents) already makes versions of that claim. v0.3's contribution is narrower and falsifiable: a controlled protocol-ablation pack on LHC tasks, with a conditional small specialist model trained only if the ablation surfaces a model-shaped residual gap.

External framing: **"The Agent State Handoff Problem."** The moment when an agent must resume from prior state — what state matters, what changed, what conflicts, what action is next — is the failure mode that breaks long-running systems. v0.3 measures it.

**Scope statement (added at reviewer's request):** v0.3 targets the handoff layer between interactive front-end agents and slower background work — what context package is sufficient for reliable continuation? This is adjacent to but narrower than the broader "interaction model" framing that posits continuous interaction, interruption, and shared context as the next paradigm (Thinking Machines, *Interaction Models*, 2025). v0.3 does not attempt to solve interaction models; it targets the auditable handoff protocol that today's agent systems and local/specialist helpers need.

## 2. Cycle structure: two phases

Phase 1 always runs. Phase 2 runs **only if** Phase 1 results clear all training gates in §5.

### Phase 1 — Protocol ablation (always runs)

H0–H4 harness conditions × N models × LHC tasks (24 v0.2 + 12 fresh held-out) × 4 gap modes × 3 trials.

**H0** Raw LHC prompt (no protocol scaffolding).
**H1** Explicit state card. Extracted programmatically from the prompt; no interpretation, no answer hints.
**H2** State card + provenance/evidence spans. Same as H1 plus span pointers into the source transcript.
**H3** State card + conflict detector. Detector flags potential inconsistencies but does **not** name the correct answer or recommend an action.
**H4** Oracle protocol. Resume artifact constructed using ground truth. **Clearly labeled as upper bound** in all reporting; excluded from any capability claim about non-oracle conditions.

**Length/recency control (H1c, H2c):** to rule out the confound that H1/H2 work *only* because they move relevant state physically closer to the probe in the prompt (a known long-context recency effect, not a state-interface effect), every Phase 1 run also evaluates two control conditions:
- **H1c** — same number of tokens as H1, but the inserted content is filler (e.g., shuffled state-card text from a different unrelated task) inserted in the same position. Same length, same recency, no semantic content.
- **H2c** — same construction for H2.

A genuine state-interface effect must show H1 > H1c and H2 > H2c by ≥ +0.10 mean overall. If the H1 − H1c (or H2 − H2c) delta is < +0.10 or has CI crossing 0, the H1/H2 effect is attributed to position/recency rather than state interface, and is **not** counted toward the T-gates in §5.

H3 inherits its control implicitly: H3 only counts toward T-gates if H3 − H2 is positive (i.e., the conflict detector adds value beyond the state card already in H2). H4 needs no control — it is upper bound only.

### Phase 2 — Conditional specialist model

If and only if all training gates in §5 fire, train `lhc-state-handoff-1.5b` (working name; final name TBD by base model selection): a small specialist that emits structured state-handoff JSON given prior context + current request.

**Output schema (frozen at training start):**
```json
{
  "active_goal": "string",
  "relevant_prior_state": "string",
  "conflicts": [
    {
      "type": "state_invalidation | decision_conflict | scope_change",
      "detail": "string",
      "evidence_spans": ["string"]
    }
  ],
  "next_action": "string",
  "evidence_spans": ["string"]
}
```

This model is **not a chat model**. It is an inference-time helper called by an agent harness around frontier LLM calls. The model card will say so explicitly.

**Training data provenance (strict):**
- Phase 2 training data is curated **only from the 24 v0.2 tasks** (the carry-forward set) and any auxiliary dev tasks authored explicitly for training (out-of-band from the LHC v0.3 evaluation pack).
- The 12 fresh held-out tasks are **excluded from training data construction** under all conditions, regardless of whether their Phase 1 traces look useful. This is enforced by a pre-training leakage check (an automated script that fails kickoff if any held-out task_id appears in the training set).
- The leakage check is run twice: once after data curation, once again immediately before training kickoff. Both runs logged and published.
- Phase 1 traces from the H4/oracle condition are also excluded from training data (independent constraint from the held-out leakage; both apply).

**Phase 2 comparison structure (precise):**
- The specialist is evaluated against same-size base on **H0 (raw)** condition on the 12 held-out tasks.
- The same-size base receives the **same JSON-schema-emit instruction prompt** the specialist would receive at inference time. Same instruction, same expected output shape.
- The same-size base does **not** receive any H1/H2/H3 scaffolding. The comparison isolates "does the specialist do better than the base at emitting the handoff JSON, with no protocol help to either side?"
- Output is scored both (a) for content quality (using the LHC v0.3 judge rubric applied to `next_action` and `relevant_prior_state` fields) and (b) for schema validity (parse rate).

## 3. Primary endpoints

- **Benchmark:** LHC v0.3, frozen at commit tagged `lhc-v0.3-frozen`. Composition: 24 v0.2 tasks + 12 fresh held-out tasks authored blind to Phase 1 results.
- **Primary gap mode:** `current` (production-relevant condition).
- **Primary metrics:**
  - LHC v0.3 **overall** mean (range 0.0–2.0, sum of correctness ∈ {0,1} + meta_awareness ∈ {0,1}).
  - LHC v0.3 **resumption** category mean.
  - **Schema validity rate** (Phase 2 only): fraction of model outputs that parse as valid handoff JSON.
- **Artifact check:** `neutral` gap mode (lorem-ipsum filler).
- **Diagnostic gap modes:** `none` and `placeholder` (run for transparency).

## 4. Comparison structure

- **Trials per cell:** 3.
- **Pairing:** all comparisons paired by (task_id, gap_mode, trial_index, harness_condition).
- **Aggregation:** mean of paired per-task scores with 95% bootstrap CI over tasks (10000 resamples, drawn at task level holding gap_mode/trial/harness fixed).
- **Variance attribution:** ANOVA-style decomposition across {model_id, harness_condition, gap_mode, task_id}. Used for kill condition K2 in §5.
- **Judge:** Claude Opus 4.7 (same as v0.2). Used for all model evaluations and the judge-stability audit.

### Phase 1 model panel

- One frontier API model: GPT-5.5 OR Claude Opus 4.7 OR Gemini 3.1 Pro (one chosen for primary; others optional based on POC-1 budget).
- One strong open 8B-class model: Qwen3-8B base (continuity with v0.2).
- One small open 1-3B model: Qwen3-1.7B or Llama 3.2-3B (final pick at Phase 1 freeze).
- Optional: Ember v0.1.5 + Qwen-local-MLX for v0.2 continuity.

### Phase 2 evaluation

If Phase 2 fires: `lhc-state-handoff-1.5b` evaluated against same-size base model on held-out tasks under H0 condition (i.e., the specialist must do its work without harness scaffolding the base model would also get).

## 5. Phase 2 training gates

Train **only if all four** hold on Phase 1 results:

- **(T1)** A non-oracle condition (H1, H2, or H3) improves mean LHC overall by **≥ +0.20** over H0, on the primary frontier model AND on the open 8B model. The condition counting toward T1 must also clear its length/recency control (Hx − Hxc ≥ +0.10, CI not crossing 0) per the H1c/H2c definitions in §2. If the only condition that meets the +0.20 bar over H0 is one whose length/recency-control delta fails, T1 is not satisfied — that effect is attributed to position/recency, not state interface.
- **(T2)** H4/oracle is **not** the only condition that helps. If H4 is the only mover, the finding is "giving the answer helps," not "harness helps" — no model is justified.
- **(T3)** ≥ **30%** of remaining errors under the best non-oracle condition are **not** solvable by the deterministic parser baseline (the v0.2 parser, or a v0.3 update of it). Verified by running parser over the residual error set.
- **(T4)** The non-oracle protocol effect (T1) survives on the 12 fresh held-out tasks, defined as: same direction, ≥ +0.10 mean improvement on held-out subset, CI lower bound > 0.

If any of T1–T4 fail: **no Phase 2.** v0.3 ships as protocol-ablation paper + dataset + updated parser. This is an honest, publishable outcome.

## 6. Phase 2 ship gates

If Phase 2 trains a model, **ship only if all five** hold on held-out tasks:

- **(S1)** `lhc-state-handoff-1.5b` beats same-size base by **≥ +0.20** mean overall on held-out state-handoff tasks (under H0 condition).
- **(S2)** 95% paired-bootstrap CI lower bound for the (specialist − same-size base) delta is **> +0.05**.
- **(S3)** Beats the deterministic parser baseline on **unstructured/free-prose** tasks. Matches or beats parser on **structured** tasks. Both held-out.
- **(S4)** Schema validity ≥ **95%** on held-out cases. (Of model outputs, 95% must parse as valid handoff JSON.)
- **(S5)** Smoke-test chat sanity: pass a 20-prompt informal chat eval at baseline-comparable quality. Not a benchmark; just confirmation that the model isn't broken on basic conversation. **Model is not marketed as chat regardless of result.**

If any of S1–S5 fail: train fails to ship. v0.3 ships as protocol-ablation paper + dataset + updated parser, plus a "we tried, it didn't clear the bar" note in the writeup. Same discipline as v0.2's Ember v0.1.5 retirement.

## 7. Benchmark integrity gates ("LHC v0.3 itself is insufficient")

Declare LHC v0.3 unfit for shipping claims (regardless of where models land) if **any** of these fire:

- **(K1)** **Judge instability** (carry-forward from v0.2 G9): rejudge a 10% audit sample with Opus 4.7 in a fresh session. If any model's overall mean shifts by ≥ **0.10** under rejudge, judge is too noisy.
- **(K2)** **Paired harness robustness fails**: for the best non-oracle harness condition (the highest mean among H1, H2, H3), the per-task (Hx − H0) paired delta must be:
  - Positive on **≥ 60%** of the 36 tasks, AND
  - Positive on **≥ 2 of 3** task categories (state_recall, resumption, planning/commitment/multi-thread grouped), AND
  - Paired bootstrap CI lower bound on the overall (Hx − H0) delta **> 0**.
  If all three conditions fail simultaneously, the harness effect is too sparse or non-robust to support a claim. Task_id explaining variance is expected and not disqualifying; sparse or unidirectional harness effects are.
- **(K3)** **Held-out collapse**: if the 12 held-out tasks show ranking inversions (model order on held-out differs from v0.2-task order by more than 2 positions), the held-out subset is not measuring the same thing as the v0.2 subset — investigate before publishing.
- **(K4)** **Manual audit miss-rate** (carry-forward from v0.2 G12): hand-audit 10% of task-trials. If ≥ **5%** of audited labels are materially wrong in a direction that would flip a per-task score, judge or rubric is unreliable.
- **(K5)** **CI overlap on harness deltas**: if 95% paired-bootstrap CIs for the (best non-oracle harness − H0) delta include deltas of **±0.15** for both the frontier and the open 8B model, benchmark lacks discriminative power for harness effects.

If any K-gate fires: stop, do not ship v0.3 model claims, publish what we have as a methodology note documenting the failure mode.

## 9. Operational protocol

### Run order

1. **Week 0 (this week):** draft model card and dataset card *as if v0.3 already shipped successfully*. Sit at `docs/v0.3/draft-model-card.md` and `docs/v0.3/draft-dataset-card.md`. Send full v0.3 plan (this doc + drafts) to external reviewer for final sign-off before kickoff.
2. **Week 1 — POC-1:** frontier model **billing pilot only**. 2 tasks × 4 gap modes × 1 frontier model, 3 trials. Purpose: extrapolate per-token costs from actual billing receipts to full Phase 1 budget. If extrapolated cost exceeds €200 for the frontier-only Phase 1 sweep, drop a model from the panel or reduce trials. **No capability kill condition fires from Week 1 alone.** A frontier ceiling observation on 2 tasks is too small for a hard fork. If the billing pilot also incidentally suggests frontier ceiling behavior, log it and revisit after Phase 1 sweep completes (see §9.5 below). Protocol ablation still proceeds — frontier ceilings do not invalidate harness effects on local/open models, which is the more practically interesting case anyway.
3. **Week 2 — Freeze H0–H4.** Code the harness conditions. Lock the protocol definitions before any full sweep. Pre-register kill conditions verbatim; no edits after this point.
4. **Weeks 3–4 — Author 12 fresh held-out tasks blind to all POC results.** Pre-budgeted calendar time, not "I'll get to it." Authoring done by Hugo, not by Claude or any model; tasks must be original scenarios (no transformations of v0.2 tasks). Each held-out task must pass an **anti-isomorphism checklist** before acceptance:
   - **(I1) Different domain:** the scenario domain (kitchen / ATC / NICU / court / library / coding / etc.) must not match the domain of any v0.2 task.
   - **(I2) Different state-shape:** the structure of the loaded state (list of items / time-conditional rule / multi-actor protocol / counter / etc.) must not be a one-to-one rename of a v0.2 task's structure. If a v0.2 task uses "86 until 7pm" time-conditional rule, the new task may use a time-conditional rule but with different scaffolding (e.g. "Pause Stage 3 until ECG steady"), not a renamed version.
   - **(I3) Different distractor pattern:** the gap content category (procedural chatter / unrelated news / similar-topic distractor) must be drawn fresh for each held-out task, not lifted from v0.2.
   - **(I4) No solved-by-rote:** if a v0.2 task is solved by the deterministic parser via a specific schema lookup, the new task should not be solvable by the same lookup. Run the parser against the new task before acceptance; if parser scores ≥ 1.5 mean overall, reject the task as too parser-trivial unless we explicitly want a structured task in the held-out set.
   - **(I5) Written declaration:** for each held-out task, Hugo writes a one-paragraph "why this is not isomorphic to any v0.2 task" note saved at `evals/v0.3/held_out/<task_id>/isomorphism-note.md`. The note is committed before any sweep run. Reviewer or third party can audit.
5. **Weeks 5–6 — Full Phase 1 sweep.** Run all conditions × all models. Compute T-gates and K-gates via a single analysis script. The script's output IS the verdict.
6. **Phase 1 decision per §8.** If Phase 2 fires: continue. If not: ship Phase 1 artifacts.
7. **Weeks 7–9 (conditional) — Phase 2 training.** Curate training data from Phase 1 traces (with strict no-oracle leakage check). Train. Evaluate against S-gates on held-out tasks.
8. **Phase 2 decision per §8.** Ship or don't ship the model.

### Post-sweep adjustments (§9.5)

After Phase 1 sweep completes, if the frontier model ceilings on `current` gap mode (overall mean ≥ 1.85, paired bootstrap CI width ≤ 0.15) across the full 36-task set, do the following — **without** retroactively editing this DECISION doc:

- Note in the published results that LHC v0.3 is at-or-near ceiling for the frontier model. This is a known limitation, not a benchmark failure.
- Continue evaluating protocol ablation on the open 8B and small 1-3B models. The harness effect is **measured on whichever models have headroom**; if the frontier ceilings, the open models become the primary capability-axis comparison.
- Add a §1.1 to the published v0.3 writeup acknowledging the frontier ceiling and flagging "build harder tasks for v0.4" as a carry-forward.

A frontier ceiling is **not** a K-gate trip. It changes the framing of the result, not its validity.

### What we will NOT do

- **No goalpost adjustment.** If gates don't fire as expected, we land in the appropriate decision-matrix row, not a loosened threshold.
- **No selective reporting.** All scorecards published, all gate computations published.
- **No pre-Phase-2 data curation tricks.** Training data for Phase 2 is curated from Phase 1 traces using a deterministic recipe pre-registered alongside the H0–H4 protocol freeze. No after-the-fact data selection.
- **No held-out task contamination.** The 12 fresh tasks are authored before Phase 1 sweep results are visible. If schedule slips and tasks haven't been authored when sweep is ready, sweep waits.
- **No Clayer in published framing.** v0.3 is independent research. Internal use of v0.3 artifacts in Clayer is fine; cross-linking publicly is not.

### Budget cap

- **Hard ceiling:** €2000 EUR all-in.
- **Expected spend:** €100–500 (POC-1 API + minor GPU rental for Phase 2 training if it fires).
- If projected spend exceeds €1000, pause and re-evaluate before continuing.

## 12. Glossary

- **paired bootstrap CI**: standard nonparametric CI. Resample tasks with replacement 10000 times, compute paired delta on each resample, take 2.5th and 97.5th percentiles.
- **harness condition**: the H0–H4 protocol layer applied to the LHC prompt before model inference. H1c and H2c are length/recency-matched control conditions for H1 and H2 respectively.
- **non-oracle condition**: H1, H2, or H3. Uses extracted/computed information from the prompt; does not embed ground truth.
- **oracle condition (H4)**: uses ground-truth state/conflicts/actions to construct the resume artifact. Upper bound only.
- **length/recency control (H1c/H2c)**: same token count and prompt position as H1/H2 but with semantically irrelevant filler (shuffled content from an unrelated task). Used to attribute observed effects to state interface vs. position/recency.
- **scorecard**: one JSON file per (model × harness × gap_mode × trial_index) cell, containing scored task-trials for all tasks in scope. Follows v0.2 convention.
- **task-trial**: one (task × trial) row within a scorecard. Phase 1 has 6480 task-trials total at minimum panel size; Phase 2 has 432 task-trials.
- **schema validity rate**: fraction of model outputs that parse as valid handoff JSON per the §2 schema.
- **materially wrong** (K4): a judge label that, after reading the full prompt + response, the human auditor would assign differently — at least one of correctness or meta_awareness flips.
- **anti-isomorphism checklist** (§9 Run order step 4): the I1–I5 constraints a held-out task must satisfy to be admitted into the held-out subset.
